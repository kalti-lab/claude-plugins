# kalti 연구일지 작성

kalti는 연구 모임이다. 멤버는 각자 한 일을 **연구일지**로 남기고, 운영자가 그 일지들을 모아 지식(온톨로지)으로 정제한다. 이 스킬은 **일지를 쓰는 쪽**의 규약이다.

핵심 정신: 멤버는 **일지만 쓰면 된다.** 가설·발견을 지식 그래프로 엮는 일은 운영자가 따로 한다. 그러니 일지를 쓸 때는 "나중에 내가 없어도 이 글만 보고 똑같이 재현할 수 있게, 한 일을 그대로" 남기는 데만 집중한다. 형식을 지키는 이유는 깔끔함이 아니라 — 일지가 정제 단계에서 기계적으로 읽히고, 볼트 그래프에 정확히 연결되기 때문이다.

## 두 층 구조 (왜 이렇게 쓰나)

볼트는 두 층이다. 일지를 쓸 때 이 경계를 지키는 게 가장 중요하다.

- **일지 층 `journals/<이름>/`** — 멤버가 쓰는 작업 기록 = **증거**. 네가 쓰는 곳.
- **온톨로지 층 `ontology/`** — 일지에서 정제해 뽑은 "계속 살아있는 지식" 객체 = 운영자만 쓴다. **직접 만들거나 고치지 않는다.**

가설이나 발견이 떠오르면 **일지 본문에 그냥 적어둔다.** 운영자가 정제할 때 그걸 골라 온톨로지 카드로 올린다. 더 챙기고 싶으면 `<이름>/proposals/`에 한 줄 메모를 남겨도 된다.

## 어디에 · 파일명

- 본인 폴더 `journals/<본인 이름>/`(볼트 최상위 `journals/` 아래) 에 새 노트를 만든다. 예: `journals/aram/샘플러별-디테일-비교.md`
- **파일 이름 = "무엇을 했는지" 드러나는 한글 제목.** 날짜를 파일명으로 쓰지 않는다(`2026-06-18.md` ❌). 날짜는 프론트매터 `date`에 적는다. 제목으로 파일명을 지어야 나중에 사람도 그래프도 한눈에 찾는다.
- 남의 폴더(`journals/<다른사람>/`)나 `ontology/`·`_shared/`에는 쓰지 않는다.

## 맨 위 7칸 (프론트매터)

노트 맨 위 `---`와 `---` 사이에 이 7칸을 채운다. 그대로 복사해서 값만 채워라:

```yaml
---
id:        # 타입약어-슬러그 (예: exp-sampler-detail). 한 번 쓰면 재사용 안 함
title:     # 무슨 작업인지 한 줄
date:      # YYYY-MM-DD
author:    # 본인 (폴더 이름과 같게)
type:      # 아래 type 목록에서 하나
tags: []   # 아래 tags 허용 목록에서만
project: "[[이미지생성-파이프라인]]"   # 어느 종목에 속하는지 (종목 노트의 파일 이름)
---
```

가설을 검증하는 실험이면 한 칸 더 붙인다: `tests: "[[가설 노트 이름]]"`

`id` 규약: `타입약어-슬러그`. 약어는 type별로 — `exp-`(experiment) / `invest-`(investigation) / `build-`(build) / `read-`(reading) / `meet-`(meeting) / `decide-`(decision) / `retro-`(retro). 날짜·순번을 id에 넣지 않는다.

## 링크는 파일 이름으로 (id 아님)

`id`는 글마다 붙는 고정 표지일 뿐이다 — 다른 글이 이 글을 가리킬 때의 식별자. **위키링크 `[[ ]]`는 항상 그 노트의 "파일 이름"으로 건다. id로 걸지 않는다.** Obsidian이 링크를 파일 이름(basename)으로 해석하기 때문이다. 예: `project: "[[이미지생성-파이프라인]]"` (O), `project: "[[proj-image-pipeline]]"` (X).

## type — 일지 종류 (하나만)

| 값 | 뜻 |
|---|---|
| `experiment` | 실험: 조건 정해 결과 측정 |
| `investigation` | 조사: 원인·현황 파악 |
| `build` | 구현: 코드·환경·도구 만들기·설정 |
| `reading` | 자료정리: 논문·문서 읽고 정리 |
| `meeting` | 회의 기록 |
| `decision` | 결정 기록 |
| `retro` | 회고 |

## tags — 정해 둔 단어만

`infra` · `security` · `storage` · `network` · `ai` · `data` · `tooling` · `report` · `diffusion` · `sdxl` · `sampling` · `image` · `prompt`

목록에 없는 새 태그는 팀과 합의한 뒤 추가한다. 제멋대로 만들지 않는다 — 태그가 제각각이면 정제·검색이 깨진다.

## 본문 — 이 순서로 (7섹션)

```markdown
# (제목)

## 질문 / 목적
무엇을, 왜.

## 배경
이어지는 맥락.

## 한 일 (방법)
따라 할 수 있게. 수치·명령·버전·경로 그대로.

## 결과 / 관찰
나온 그대로. 실패·이상치도 빼지 말 것.

## 해석 / 막힌 점·해결

## 결정
무엇을, 왜.

## 다음 액션
```

실험·조사·구현 등 어떤 type이든 이 골격을 쓴다. 해당 없는 섹션은 비워도 되지만 순서·제목은 유지한다 — 정제 단계가 이 구조를 전제로 읽는다.

## 다섯 가지 원칙

- **재현 가능성**: 내가 없어도 이 글만 보고 똑같이 따라 할 수 있게. "적당히" ❌ → "값 12로" ⭕
- **추적성**: 어떤 결론이든 근거를 이 글 안에서 짚을 수 있게.
- **하면서 바로 적기**: 끝나고 몰아 쓰면 수치·과정이 빠진다.
- **실패도 빠짐없이**: 안 된 것·이상한 것도 남겨야 같은 삽질을 반복하지 않는다.
- **지우지 말고 고치기**: 틀린 내용은 지우지 말고 취소선(`~~틀린 내용~~`)이나 덧붙임으로, 이유와 함께.

## 채워진 예시 (실험)

```markdown
---
id: exp-sampler-detail
title: 샘플러별 마무리 디테일 비교
date: 2026-06-15
author: aram
type: experiment
tags: [diffusion, sdxl, sampling]
project: "[[이미지생성-파이프라인]]"
tests: "[[가설-샘플러가-디테일을-좌우한다]]"
---

# 샘플러별 마무리 디테일 비교

## 질문 / 목적
SDXL 포트레이트에서 샘플러를 바꾸면 머리카락·피부 마무리 디테일이 실제로 갈리는가.

## 배경
스텝 수만으로는 디테일이 안 올라간다는 직전 결과([[스텝수-디테일-실험]]) 이후, 변수를 샘플러로 옮겨 본다.

## 한 일 (방법)
- 고정: step 30, CFG 6.5, seed 3개(12/4521/88012), Karras 스케줄.
- 변수: 샘플러만 — Euler a / DPM++ 2M / DPM++ 2M SDE.

## 결과 / 관찰
DPM++ 2M SDE에서 속눈썹·머리카락 끝 가닥 분리가 가장 또렷. Euler a는 뭉개짐.

## 해석 / 막힌 점·해결
마무리 디테일은 스텝보다 샘플러(특히 SDE 계열)가 좌우한다. Karras 스케줄과 맞물린 노이즈 거동 차이로 보인다.

## 결정
포트레이트 프리셋 샘플러를 DPM++ 2M SDE + Karras로 고정.

## 다음 액션
업스케일 단계에서도 같은 우위가 유지되는지 별도 검증.
```

## 일반 일지 골격 (조사·구현·자료정리·회의·결정·회고)

프론트매터에서 `type`을 바꾸고 `tests`를 빼면 된다(가설 검증이 아니므로). 본문 7섹션은 동일.

```yaml
---
id:               # invest- / build- / read- / meet- / decide- / retro-
title:
date:             # YYYY-MM-DD
author:
type:             # investigation | build | reading | meeting | decision | retro
tags: []
project: "[[종목 노트 이름]]"
---
```

## 하지 말 것 (요약)

- 온톨로지 파일(`ontology/`)을 직접 만들거나 고치지 않는다 — 가설·발견은 일지 본문에 적고 운영자 정제에 맡긴다.
- 남의 폴더에 쓰지 않는다.
- 파일명을 날짜로 짓지 않는다.
- 허용 목록 밖 태그를 만들지 않는다.
- 링크를 id로 걸지 않는다.
