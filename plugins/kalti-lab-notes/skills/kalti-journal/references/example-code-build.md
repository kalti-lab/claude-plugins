# 채워진 예시 — 코드 작업 (investigation/build)

코드를 만지는 일(파라미터 튜닝, 봇·파이프라인 수정, 버그 추적)은 일지가 **노이즈로 가장 쉽게 무너지는** 종류다. 커밋 해시, 수정한 파일 목록, 함수명과 줄번호, DB 컬럼, env 키 이름, PID, 내부 변수명을 그대로 옮겨 적기 쉬운데 — 이것들은 코드와 git이 이미 보관하는 "어디를 어떻게 눌렀나"이지 발견이 아니다.

아래는 **같은 작업**을 두 번 적은 것이다. 먼저 흔히 빠지는 **나쁜 예(노이즈)**, 그다음 **정리한 좋은 예**. 좋은 예에는 해시·파일목록·함수명/줄번호·PID·env키·`·`/`→` 밀집이 하나도 없지만, 재현에 필요한 값과 발견·결정은 그대로 남는다는 점을 본다.

저장된 위치 예시 — `journals/jinsik/거래봇/20260625-과매도-진입필터-손절규칙.md`.

>

무엇이 문제인가 — 함수명·줄번호·파일목록·DB명·env 키·해시·PID·`/`·`→` 밀집이 모두 "키 입력 로그"다. 다른 팀원은 `bidx`도, `check_oversold_buy`도, `13bb677`도 읽을 수 없다.

```markdown
# 과매도 매수의 진입필터·손절 규칙 진단

## 질문 / 목적
과매도 신호로 매수해 일정 수익에서 익절하는 전략의 약점을 실제 거래 기록으로 진단한다.
승률과 기대수익을 끌어올릴 진입 조건과 손절 규칙을 찾는다.

## 배경
직전 분석([[20260623-트랙별-승률분석]])에 이어, 이번엔 진입 시점의 거래량과 손절 타이밍을 본다.
원칙: 기록된 실거래를 우선 쓰고, 없는 구간만 일봉으로 보완한다.

## 한 일 (방법)
- 신호일 거래량을 직전 20거래일 평균으로 나눈 값을 "거래량배수"로 정의한다.
  신호일은 매수 전 거래일 — 봇이 장 시작 전 실행되어 전일 종가까지만 확정된 봉을 본다.
- 진입 거래량배수 상한을 0.8~1.1 사이에서, 손절은 보유 8·10·12거래일과 미실현 -8·-10·-12%를 교차해 시뮬한다.
- 손절 시뮬은 매수 다음날부터 매도일까지 일별로 재생해, 손절선과 익절선 중 무엇을 먼저 건드리는지로 판정한다.

## 결과 / 관찰
- 실거래 청산 94건: 실현승률 83%, 평균 +3.0%, 최악 -28.4%.
- 거래량배수 1.0 미만으로 진입을 거른 48건: 승률 93.8%, 평균 +4.9%.
  상한을 0.8~1.1로 옮겨도 승률 89~95%로 평평해 단일 값 요행이 아니다.
- 시간 손절(보유 10거래일·미실현 -12%): 승자를 자르지 않으면서 최악 손실을 -28.4%에서 -19.5%로 줄였다.
- 가격만 보는 타이트한 손절(-3~-7%)은 회복 전 승자까지 잘라 오히려 손해였다.

## 해석 / 막힌 점·해결
- 과매도 진입은 평균적으로 깊게 눌렸다 회복한다 — 그래서 가격 손절이 아니라 "시간이 지나도 미회복"이 맞는 신호다.
- 좁은 표본에서 좋아 보인 조건이 더 넓은 표본에선 익절 건까지 걷어내 효과가 사라졌다(인샘플 착시).
- 14일 전체 저점을 쓰는 기록값은 익절 이후 낙폭까지 포함해 손절 판정엔 못 쓴다 — 경로 재생으로만 가능.

## 결정
- 진입 거래량배수 상한 1.0, 시간 손절 보유 10거래일·-12%, 익절 목표 +5% 유지로 봇에 반영.
- 진입 종목의 거래량배수를 매수 시점에 함께 기록해 다음 기간 재검증에 쓴다.

## 다음 액션
- [ ] 이번 진단은 약 2개월 인샘플 — 누적 후 다른 기간으로 재검증.
- [ ] 기존 보유 종목이 새 시간 손절 규칙에 걸리는지 다음 매도 점검에서 확인.
```

핵심 — 결과를 좌우하는 **값**(거래량배수 1.0, 보유 10거래일, -12%, +5%, 승률 93.8%)은 그대로 남기되, 그 값을 **무슨 파일의 무슨 함수를 고쳐** 넣었는지는 적지 않는다. 식별자는 "신호일 거래량을 20거래일 평균으로 나눈 값"처럼 뜻으로 풀어 쓴다.
